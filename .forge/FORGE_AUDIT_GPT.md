# GPT Independent Audit Report
## Findings
| # | Severity | Category | File | Line | Issue | Recommended Fix |
|---|---|---|---|---:|---|---|
| 1 | Critical | Security | src/server.ts | 23 | Token is not verified or resolved server-side. The API trusts client-supplied `customer_id`, `account_name`, and `segment`, so any caller can spoof any account. | Accept only `token`, `topic`, and `body`; verify HMAC/JWT/opaque token server-side; derive customer/account/segment from verified claims; reject expired, tampered, or reused tokens. |
| 2 | Critical | Security | src/server.ts | 45 | Admin/data endpoints are unauthenticated: DLQ reprocess, aggregation, trends, at-risk accounts, and health are public while the server binds to `0.0.0.0`. | Add authentication and authorization for all dashboard/admin APIs; keep only the feedback form endpoint public; protect admin POSTs with CSRF or bearer auth. |
| 3 | High | Availability | src/server.ts | 23 | Public ingestion has no rate limiting, idempotency, or meaningful length limits, allowing DB/queue growth and future LLM cost amplification. | Add Fastify JSON schema validation with `maxLength`, lower `bodyLimit`, rate-limit by token/IP, and enforce one submission or idempotency per feedback token. |
| 4 | High | Reliability | src/queue.ts | 47 | Claimed jobs move to `processing` with no lease, heartbeat, or stale-job recovery. A crash/redeploy after claim leaves feedback unprocessed forever. | Add `locked_at`/`lease_expires_at`; reclaim expired `processing` jobs on claim/startup; expose recovery for stale processing jobs, not only `dead` jobs. |
| 5 | High | Concurrency | src/queue.ts | 50 | Job claim is `SELECT` then `UPDATE` without a transaction or conditional guard, unsafe with two app instances. Stale workers can also overwrite completed jobs via `failJob`. | Use atomic `UPDATE ... WHERE status='queued' ... RETURNING *`; include `status`, `attempts`, and `locked_by` guards in `completeJob`/`failJob`; wrap extraction insert plus completion in one transaction. |
| 6 | High | External Calls | src/worker.ts | 24 | LLM extraction has no timeout or cancellation. A hung provider call can stall the worker and any `/api/aggregate` request indefinitely. | Add SDK request timeout or `AbortController`; fail the job on timeout; make timeout configurable and tested. |
| 7 | High | Retry Logic | src/queue.ts | 67 | Retries happen immediately in a tight drain loop. A transient provider outage dead-letters all queued feedback after three rapid attempts. | Add `run_after` with exponential backoff and jitter; distinguish retryable provider failures from permanent validation failures. |
| 8 | Medium | Architecture | src/server.ts | 51 | `/api/aggregate` drains the full extraction queue synchronously inside the HTTP request, mixing trend aggregation with slow LLM work. | Aggregate already-processed submissions only; trigger queue draining as a background job; return `202` for long-running admin actions. |
| 9 | Medium | Spec Compliance | src/server.ts | 25 | `topic` is arbitrary text despite the requirement for a dropdown. Attackers or client bugs can pollute theme aggregation. | Define allowed topics in code and DB with a `CHECK` or reference table; validate `topic` in the request schema. |
| 10 | Medium | Spec Compliance | public/index.html | 45 | No customer feedback form or tokenized link route exists; only the leadership dashboard and raw POST API are implemented. | Add `GET /feedback/:token` serving a form with topic dropdown and free text; submit only token/topic/body to the API. |
| 11 | Medium | Resilience | src/aggregate.ts | 138 | Aggregation timer has no error boundary, and `JSON.parse` failures in aggregate data can crash the process. Worker interval also has no catch around `drainQueue`. | Wrap timer callbacks in `try/catch`; log and metric failures; validate JSON at write time or with DB checks; quarantine corrupt rows. |
| 12 | Medium | Production Config | src/extractor.ts | 118 | Missing `ANTHROPIC_API_KEY` silently switches to mock extraction, so production can run without LLM analysis while appearing healthy. | Require explicit `EXTRACTOR_MODE=mock` for local/test; fail startup in production without a real provider key. |
| 13 | Low | Supply Chain | package.json | 13 | Dependencies use broad `^` ranges and there is no CI/script enforcing lockfile install or audit. | Use exact dependency pins or enforce `npm ci`; add `npm audit --omit=dev` or equivalent CI dependency scanning. |

## Summary
Total findings: 13, with 2 Critical, 5 High, 5 Medium, and 1 Low. Top risk area is the public trust boundary: the system trusts client identity, exposes admin/business data without auth, and allows unauthenticated queue/LLM-cost actions. Spec coverage is partial: the durable write-first skeleton, worker, extraction schema, DLQ, and aggregate dashboard are present, but token resolution, the actual feedback form, production-safe retries, crash recovery, and admin protection are missing. Production readiness is low without fixing auth/token verification, job leasing, retry backoff, LLM timeouts, and endpoint hardening.
